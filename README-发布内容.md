# 小学二年级联合会内容中心：发布说明

本项目是纯前端静态网页，可部署到 GitHub Pages、Cloudflare Pages、Netlify 或普通静态空间。请通过 **HTTPS**（或本机 `localhost`）访问；直接双击 `file://` 打开时，浏览器可能阻止读取内容文件，Service Worker 与 Web Crypto 也无法完整运行。

## 加密文件包含什么

由“发布我的”生成的 `.txt` 文件中，以下信息全部位于 AES-256-GCM 加密数据内部：

- 内容类型（文章或帖子）
- 文章标题
- 内容摘要
- 创作者名称
- Markdown 正文

加密文件不再把自动获取的完成日期或完成时间写入密文内部。网页本地保存成稿时仍会保留并显示日期；正式发布时，请在 `content/files.json` 中用 `title`、`name`、`data` 记录列表所需的标题、列表创作者名称和创作日期。

下载文件名会直接使用标题，例如：

```text
欢迎来到内容中心.txt
```

## 发布文章或帖子的正确方法

1. 打开 `community.html`，进入“发布我的”。
2. 填写标题、选择文章或帖子、输入 Markdown 内容。
3. 点击“完成并保存”，填写创作者名称，下载生成的加密 `.txt` 文件。
4. 审核通过后，把该 `.txt` 文件上传到源码分支的：

```text
content/published/
```

5. 打开：

```text
content/files.json
```

6. 按下面格式登记标题、列表创作者名称和创作日期。`articles` 里的项目会出现在“官方文章”，`posts` 里的项目会出现在“官方帖子”；每个项目只使用 `title`、`name`、`data` 三个字段：

```json
{
  "articles": [
    { "title": "欢迎来到内容中心", "name": "小学二年级联合会", "data": "2026-06-24" }
  ],
  "posts": [
    { "title": "一次简单的规律观察", "name": "小学二年级联合会", "data": "2026-06-24" }
  ]
}
```

提交并推送源码后，网页打开列表时只读取 `content/files.json`，不会提前读取或解密正文文件。点击进入某篇文章或帖子时，网页才按标题推导同名 `.txt` 文件并读取 `content/published/` 中的加密文件，再从密文内部取得内容类型、真实创作者名称、摘要和 Markdown 正文；详情页中的创作者名称以加密文件内部为准。日期从 `files.json` 的 `data` 字段读取，用于列表展示和排序。

## 为什么仍需登记清单

GitHub Pages 等普通静态网站不能让前端 JavaScript 自动列出服务器目录中的全部文件。因此，纯前端方案至少需要一份内容清单。这个项目的清单仅保存列表所需的标题、列表创作者名称和日期，不保存正文内容。

## 隐藏或删除内容

- 临时隐藏：从 `content/files.json` 中删除对应的 `title`、`name`、`data` 条目，但可以保留 `.txt` 文件。
- 完全删除：同时删除 `content/files.json` 中的清单条目和 `content/published/` 中的同名 `.txt` 文件。
- 替换文件：上传新的 `.txt`，并把清单中的 `title`、`name`、`data` 更新为新内容。

编辑清单时请注意：`title` 应与下载得到的 `.txt` 文件名主体一致；如果标题包含文件名不支持的符号，页面会使用与下载时相同的安全文件名规则推导 `.txt` 文件名。

## 内容刷新

“官方文章”和“官方帖子”右上角都有“手动刷新”。按钮会绕过普通列表缓存，重新读取 `content/files.json`；正文加密文件仍会在进入详情页时按需读取。

Markdown、DOMPurify、KaTeX 及本地静态资源最长缓存 24 小时；草稿和本地成稿保存在当前浏览器中。


## 中文编码说明

本项目的网页、脚本、清单和说明文档统一采用 **UTF-8**。发布加密 `.txt` 时也会写入 UTF-8 BOM，网页读取 `files.json` 和加密文件时会强制按 UTF-8 解码，不依赖服务器可能错误的默认编码。

请不要使用“ANSI”或 GBK 编码另存 `content/files.json`。推荐编辑器设置为 `UTF-8 with BOM` 或 `UTF-8`。

## 加密边界

下载文件使用浏览器 Web Crypto 的 AES-256-GCM，并通过 PBKDF2-SHA-256 派生密钥。因为网站是纯前端并需要自行解密公开内容，解密密钥逻辑必须存在于前端源码中。因此，它能避免文件以普通明文直接显示，并提供加密完整性校验，但不能替代服务器端机密存储。
